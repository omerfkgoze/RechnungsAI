## What Went Well

1. **Versioned format architecture realized** — Winston's Epic 4 retro recommendation (`formats/extf-vXXX.ts` pure functions) is now load-bearing. The `packages/datev` shape means EXTF v710/v720 = new file, zero diff to existing code. This is the architectural payoff of two epics of discipline.

2. **Encoding decision over epic spec** — Spike P1 overrode the epic's "Windows-1252" line in favor of UTF-8+BOM. This avoided `iconv-lite` and matched the `packages/gobd` precedent. The story file made the override explicit and cited the spike — exactly the spike-first pattern A1 was designed for.

3. **Story scope reduction discipline** — All three stories opened with a "STOP — read this before reading the ACs" block:
   - 5.1: "tiny — one column + tests + indicator" (vs the epic's broader implication)
   - 5.2: "pure compute, no web app"
   - 5.3: "wire-up only, email descoped to Epic 8"
   This prevented Story 1.5–style sprawl and kept review patches in scope.

4. **GOZE's smoke-test concern produced architecture** — On Story 5.3, GOZE flagged: *"kullanıcı export erstellen'e bastı, faturalar exported oldu, kullanıcı pencereyi kapatınca bir daha indiremiyor."* This became review decision D1, which became P16 (TTL 1h→24h), P17 (`<LastExportCard>`), and P18 (dialog close-lock). The bug class (state mutation before user-visible completion) was caught at user-empathy level, not technical level. Smoke-test-as-design-review is the new normal.

5. **Atomicity bug caught at review** — Story 5.3 patch P1 found a non-atomic insert→status-flip→audit chain with three concrete failure modes (concurrent calls, failed UPDATE, partial transition). Resolution: `commit_datev_export` Postgres RPC. This is exactly the GoBD compliance surface where atomicity matters, and the review caught it before merge.

6. **`mapBuSchluessel` reuse** — Story 5.2 imported the shared helper instead of reimplementing the rate-to-Schlüssel mapping. Story 3.3's investment compounded.

7. **Shared filename helper extraction** — Story 5.3 didn't copy-paste `toTenantSlug` from the archive route; it extracted it during the story (P10 then hardened the German-character handling). Two routes share one source of truth — the right way to refactor as you go.

## Challenges

### Story 5.3 Patch Spike (19 patches + 2 post-merge fixes)

Worse than Story 4.3's 18-patch spike. Root causes:

- **Four new surfaces in one story** — Server Action + RPC + Route Handler + Dialog + Migration + Wrapper + Filename helper. Even with the wire-up framing and pattern citations, the surface count exceeded what a single spike covered. Story 4.3 had 4 surfaces / 18 patches; Story 5.3 had 6 surfaces / 19 patches. The pattern is consistent.
- **No dedicated spike for the wire-up** — Spike P1 covered the format, not the dialog+action+route choreography. The "this is just wire-up" framing was honest but underestimated the integration complexity.
- **Atomicity surface only visible at review** — The race conditions between insert/update/audit were not obvious in story planning. They emerged from adversarial review (Blind Hunter found three failure modes).
- **GOZE's recovery concern only emerged from smoke-test thinking** — The "user closes window, can't re-download" failure mode is invisible from a story spec's perspective. Required user-empathy that the spec didn't carry.

### Post-Merge Fixes — Code Review Did Not Catch Everything

Two `fix:` commits landed after `done story 5-3`:
- `789afa2 fix: datev async, nativeButton issues` — async/Promise handling and `<Button>` native-element issues survived 19 review patches
- `9d0094b fix: last-export-card mobile UI issue` — the newly-added `LastExportCard` (itself born from review patch P17) had mobile layout issues

Both caught by **GOZE's manual smoke test post-merge**. Code review (3-layer adversarial: Blind Hunter + Edge Case Hunter + Acceptance Auditor) consolidated 60+ raw findings to 26, but missed: (a) async semantics of a specific server-action call site, (b) `Button` render-prop nuance with anchor tags, (c) mobile responsive behavior of a freshly-added card. Adversarial review is strong on logic + correctness, weaker on runtime behavior in real browsers.

### Smoke-Test Status Mislabeled (P6)

Story 5.3 initial submission marked all smoke-test rows `DONE` despite the dev agent having no real browser — Epic 3 A1 / Story commitment violated. Caught by Acceptance Auditor (auditor#22) and relabeled to `BLOCKED-BY-ENVIRONMENT`. The discipline survives only with active enforcement; the create-story skill's smoke-test format template needs a sharper rule.

### `created_by` FK Blocking User Deletion (P13)

Pre-existing pattern from earlier migrations carried forward without scrutiny. NO ACTION on `created_by` blocks `auth.users` deletion (GDPR erasure scenario). Fixed to `on delete set null`. A reminder that compliance-adjacent schemas need GDPR review independent of the GoBD lens.

### Three Architectural Decisions Surfaced at Review (D1, D2, D3)

D1 (re-download surface), D2 (mailto month label), D3 (timezone semantics) were not surfaced in story planning — only by reviewers asking "what if the user…". D3 was deferred (multi-tenant TZ work), D1+D2 became patches. Story files are good at *what to build*; less good at *who fails*.

## Key Insights

1. **Smoke test surfaces architectural bugs, not just visual bugs.** GOZE's "user closes the window" concern became three code-review patches and one new dashboard component. Smoke-test-as-design-review is more valuable than smoke-test-as-QA. The format guide should explicitly invite *"could the user reach a broken state by closing / refreshing / opening a second tab?"* prompts.

2. **Wire-up stories deserve their own spike when surface count ≥ 4.** Spike P1 covered the format, not the choreography. The format spike + a separate wire-up spike (RPC + route + dialog interaction model) would have surfaced D1 and the atomicity bug before story creation.

3. **Adversarial code review is strong on logic, weaker on runtime.** The 3-layer review (Blind Hunter + Edge Case Hunter + Acceptance Auditor) consolidated 60+ findings to 26 patches. But it missed runtime issues (async handling, Button render-prop, mobile responsive) that GOZE's manual test caught. **Code review and manual smoke test are complementary, not substitutable.**

4. **Story files are weak at "who fails."** ACs are written from the happy-path actor's perspective. The "user closes the dialog mid-flight," "two tabs export the same range," "user wants to re-download tomorrow" failure modes are absent. Adding a Dev Notes section like `### Likely Failure Modes` could prompt this thinking before review.

5. **Email infrastructure debt is being deferred for the 2nd time.** Story 5.3 had to defer email (no `@rechnungsai/email`, no `steuerberater_email` column). Epic 6 Story 6.2 will hit the same wall (correction request emails). Epic 8 Story 8.3 owns the infra but won't ship until Epic 7 is done. Decision needed: pull 8.3 forward to between Epic 6 and 7, or defer email features in 6.2 too.

6. **GDPR review is a separate axis from GoBD review.** Story 5.3 patch P13 (`created_by` ON DELETE) wasn't a GoBD concern, it was a GDPR concern. Compliance reviews are not monolithic. A migration-checklist that covers both legal regimes would be cheap.
