## Story

As the RechnungsAI engineering team,
we want to create the `categorization_corrections` database table and add `skr_code`, `bu_schluessel`, and `categorization_confidence` columns to the `invoices` table,
so that Story 3.3 (SKR Categorization and BU-SchlГјssel Mapping) has its schema ready before implementation begins and the learning feedback loop for user corrections is built on a solid foundation.

**Why now:** Story 3.3 is blocked by this schema. Without the `categorization_corrections` table and the three categorization columns on `invoices`, the Server Actions in Story 3.3 have no place to write, and the migration cannot safely run mid-story (it would break RLS and grant setup).

## Acceptance Criteria

1. **Given** the `invoices` table currently has no categorization-related columns **When** this migration runs **Then** three new nullable columns are added to `public.invoices`:
   - `skr_code text null` вЂ” the AI-suggested or user-confirmed SKR03/SKR04 account code (free-text; Story 3.3 validates format)
   - `bu_schluessel smallint null` вЂ” the DATEV tax key (8 = 7% VAT, 9 = 19% VAT, 0 = exempt, etc.); smallint chosen because valid range is 0вЂ“99
   - `categorization_confidence numeric(4,3) null` вЂ” AI confidence for the SKR suggestion, range [0, 1]; `numeric(4,3)` stores three decimal places (e.g. `0.873`) without floating-point rounding
   - All three columns are `null` until Story 3.3 populates them; no default is appropriate because null unambiguously means "not yet categorized"
   - The UPDATE grant for `authenticated` is extended to include all three new columns (following the pattern from Story 2.2 migration `20260417120000`)

2. **Given** Story 3.3 needs to track every user correction for the AI learning loop **When** this migration runs **Then** a new `public.categorization_corrections` table exists with:
   - `id uuid primary key default extensions.gen_random_uuid()`
   - `tenant_id uuid not null references public.tenants(id) on delete restrict` вЂ” RLS scoping
   - `invoice_id uuid not null references public.invoices(id) on delete restrict` вЂ” GoBD: no CASCADE delete; invoice rows are immutable
   - `original_code text not null` вЂ” the AI-suggested SKR code before the user changed it
   - `corrected_code text not null` вЂ” the code the user actually selected
   - `supplier_name text null` вЂ” denormalized from `invoice_data.supplier_name.value` at write time so learning queries can GROUP BY supplier without a join to the JSONB blob
   - `created_at timestamptz not null default now()`
   - No `updated_at` column вЂ” corrections are append-only (GoBD spirit: the correction log must not be edited)
   - No `bu_schluessel` column вЂ” BU-SchlГјssel corrections are implicit in the new `skr_code` pick; separate column deferred to Epic 5 if DATEV export reveals a need

3. **Given** tenant data isolation is non-negotiable (NFR9) **When** RLS is configured on `categorization_corrections` **Then**:
   - `alter table public.categorization_corrections enable row level security;`
   - `select` policy: `tenant_id = public.my_tenant_id()`
   - `insert` policy: `tenant_id = public.my_tenant_id()` (with check)
   - No `update` or `delete` policies вЂ” corrections are append-only
   - Grants: `grant select, insert on public.categorization_corrections to authenticated;`

4. **Given** Story 3.3 will query corrections grouped by `supplier_name` and ordered by `created_at desc` to surface the most-recent correction per supplier **When** this migration runs **Then** a composite index exists: `create index categorization_corrections_tenant_supplier_idx on public.categorization_corrections (tenant_id, supplier_name, created_at desc);`

8. **Given** `pnpm build` passes before this migration **When** this migration is added **Then** `pnpm build` still passes.

## Dev Notes

### Context from Epic 2 Retrospective

> **TD2:** `categorization_corrections` table migration вЂ” Story 3.3 cannot start without this schema.

Story 3.3 (SKR Categorization and BU-SchlГјssel Mapping) requires:
1. Columns on `invoices` to persist the AI suggestion and human override
2. A separate corrections table that acts as the training signal for the learning loop

### Schema design decisions

- **`skr_code` as `text`** вЂ” SKR codes are strings like `"3400"`, `"4600"`. A numeric type would require casting in every query and prevent future codes that might include letters. Text + application-layer validation in Story 3.3.
- **`bu_schluessel` as `smallint`** вЂ” DATEV BU-SchlГјssel are single or double-digit integers (0вЂ“99). smallint (2 bytes) is sufficient and cheaper than integer.
- **`categorization_confidence` as `numeric(4,3)`** вЂ” avoids the `0.1 + 0.2 = 0.30000000000000004` problem for confidence scores. Precision 4, scale 3 = values like `0.873` or `1.000`.
- **`supplier_name` denormalized in corrections table** вЂ” avoids a `->>`-path JOIN into the `invoice_data` JSONB blob on every learning query. Trade-off: if the user later edits `supplier_name` via Story 3.2, the denormalized value in corrections becomes stale вЂ” acceptable because the learning algorithm uses it as a fuzzy grouping hint, not an exact key.
- **Append-only corrections** вЂ” no UPDATE/DELETE grants. Matches GoBD spirit and keeps the learning history auditable.

### Migration naming

Pattern from existing migrations: `YYYYMMDDHHMMSS_<description>.sql`. Today = 2026-04-21, suffix `000000` (no other migrations today yet вЂ” verify with `ls supabase/migrations/` before writing).

### What NOT to do in this task

- Do NOT add `Zod` schemas or TypeScript types вЂ” Story 3.3 owns those
- Do NOT create Server Actions вЂ” Story 3.3 owns those
- Do NOT run `supabase db reset` (that would wipe GOZE's local data) вЂ” only `supabase migration new` or manual file creation

### Patterns from existing migrations

- `public.my_tenant_id()` SECURITY DEFINER helper used for all RLS policies (see `20260415000000_fix_rls_recursion.sql`)
- `on delete restrict` for GoBD-relevant foreign keys (see `20260417100000_invoices_table.sql`)
- Grant structure: explicit column-level UPDATE grants for `authenticated` (see `20260417120000_invoices_extraction_columns.sql`)
- `extensions.gen_random_uuid()` for primary keys (Supabase convention вЂ” not `gen_random_uuid()` unqualified)
