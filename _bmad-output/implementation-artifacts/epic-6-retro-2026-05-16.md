## What Went Well

1. **Multi-session discipline model scaled to 6 sessions.** Each session opened with a clear pickup point from dev notes and closed with an explicit deferred-work tracker. No context rebuilding waste between sessions. The approach first used informally in earlier epics became a repeatable pattern here.

2. **224-rule coverage linchpin held throughout.** `rules.coverage.test.ts` asserted every manifest ID present in the union of rule arrays (real or typed no-op stub) every session. Rules were never silently lost — the linchpin made drift visible before merge.

3. **MIT-only license discipline enforced at spike stage.** P2 spike rejected `@stackforge-eu/factur-x` (EUPL-1.2 SaaS-triggering) before any code was written. `pdf-lib` + hand-rolled name-tree walker shipped. Zero license risk.

4. **`projectToInvoiceData` AI bypass — correct architectural choice.** Valid e-invoices short-circuit AI extraction entirely: `confidence: 1.0` from structured XML, status goes directly to `ready`. Token savings + higher accuracy. D5 decision from P4 spike paid off.

5. **6.2 low patch count (D=1, P=2) despite non-trivial surface.** UI card + Server Action + mailto helper + migration. Pattern citations (`buildSteuerberaterMailto`, `nativeButton={false} render={<a>}`, `compliance-warnings-banner.test.tsx`) worked as anchors. Date parsing bug (datetime → garbled supplier date) caught at review before merge.

6. **GOZE's routine pre-review check caught UX cross-interference.** When `packages/validation` was wired in, the invoice detail page showed `extraction_error` for validation-failed invoices and the new validation card + fields weren't visible. GOZE caught this during in-progress before review. Demonstrates that owner-level UX attention catches class of issues that automated tests miss.

## Challenges

### 1. Story 6.1 Scope Larger Than Spike Estimated

Spike P1 said "~150 rules." Actual EN 16931 coverage required 224 IDs across 6 sessions. The spike gave a single point estimate, not a range. Mid-story, "will this session finish?" uncertainty arose in multiple sessions.

Root cause: EN 16931 rule count is deterministic (it's a published standard), but the spike didn't distinguish between BR-* core rules and the full set including BR-CO-*/BR-DEC-*/BR-CL-*/VAT-matrix/CIUS. The full scope was only visible after vendoring the KoSIT manifest.

### 2. Mock-Chain Tests Deferred 5 Sessions

AC #31 explicitly requested 9 mock-chain integration cases in `invoices.test.ts`. These were deferred to Session 6 with the rationale "coverage at pure-helper layer via `validation-helpers.test.ts`." Helper coverage was necessary but not sufficient — the actual Supabase mock-chain setup in `invoices.test.ts` required a separate onboarding effort at Session 6. The delay created a window where the action's real integration behavior was unverified.

### 3. New Feature ↔ Existing UI Cross-Interference

When `packages/validation` was integrated, `invoice-detail-pane.tsx` still read `extraction_error` for status display — so validation-failed invoices appeared as "extraction errors." The new validation card and columns were also invisible until the page query was updated. Both caught by GOZE during in-progress.

Root cause: Story spec described what to add but not which existing components depended on the same data paths. No "Affected Existing UI" checklist existed.

### 4. A4 GDPR Discipline Lapsed

Epic 5 A4 committed to a GDPR migration review question per migration. Neither 6.1 nor 6.2 migration review notes show this question asked explicitly. No `auth.users` FK was added (no practical harm), but the process commitment was not followed.

## Key Insights

1. **Multi-session scope requires min/max estimates, not point estimates.** "~150 rules" created false confidence. "minimum 120 / maximum 250 depending on VAT matrix and codelist rules" would have set honest expectations. Spikes for standards-based work should derive min/max from the published specification sections.

2. **Integration tests belong to the session that implements the feature, not the last session.** Mock-chain tests for `extractInvoice` were the same session's work as the XML branch logic. Deferring them deferred discovery of setup complexity. Rule: if an AC names a test file and case list, those tests are in-scope for the session that writes the corresponding code.

3. **Owner-level UX review is a distinct quality gate from automated tests.** GOZE's pre-review walkthrough caught UI cross-interference that no automated test covered. This is not a failure of automated testing — it's a confirmation that the manual gate has unique value. The smoke-test-as-design-review principle (first stated in Epic 5) holds.

4. **"Etkilenen Mevcut UI" (Affected Existing UI) is a missing story spec field.** When new columns, new status values, or new packages are wired in, the story spec should list which existing components read from the same data path. A single-line field in the story template would have surfaced `invoice-detail-pane.tsx` as an affected component.

5. **Email defer (mailto shim) was correct.** A5 was decided correctly. Story 6.2 delivered the user-facing correction flow without blocking on Epic 8.3 infrastructure. The `tenants.steuerberater_email` column (prep-p3.1) is correctly scoped to Epic 8.3's transactional send, not to this story.

## P3 Review — Deferred Items (2026-05-16)

P3 migration code review'undan çıkan, Story 7.1 öncesinde ele alınması gereken iki açık madde:

### D-1 — `generated_at` UPSERT'te Güncellenmez

**Ne:** `verfahrensdokumentation` tablosundaki `generated_at` kolonu, satır ilk oluşturulduğunda `now()` değerini alır. Ancak migration'da bu kolonu UPSERT sırasında otomatik güncelleyen hiçbir mekanizma (trigger veya constraint) yoktur.

**Sorun:** Kullanıcı ilk kez Verfahrensdokumentation oluşturduğunda `generated_at = Gün 1`. Şirket ayarlarını değiştirip 10 gün sonra yeniden oluşturduğunda — UPSERT aynı satırı günceller ama `generated_at` hâlâ Gün 1 olarak kalır. Story 7.2'deki durum widget'ı bu tarihi "son oluşturulma zamanı" olarak gösterdiğinden kullanıcıya yanlış bilgi verir.

**Fix (Story 7.1 sorumluluğu):** Server Action'daki UPSERT kodu, `do update set` clause'una açıkça `generated_at = now()` eklemeli:
```sql
on conflict (tenant_id) do update
  set config_hash      = excluded.config_hash,
      pdf_storage_path = excluded.pdf_storage_path,
      generated_by     = excluded.generated_by,
      generated_at     = now()   -- ← bu olmadan stale kalır
```
ya da TypeScript tarafında UPSERT payload'ına `generated_at: new Date().toISOString()` eklenmelidir.

### D-2 — Storage Bucket Henüz Yok

**Ne:** Migration, `pdf_storage_path` kolonunu tanımlar — bu kolon, üretilen PDF'in Supabase Storage'daki yolunu tutar. Ancak migration yalnızca veritabanı tablosunu oluşturur; bu PDF'lerin yükleneceği Supabase Storage bucket'ı ayrıca oluşturulmalıdır.

**Sorun:** Story 7.1 PDF upload kodu çalıştığında hedef bucket yoksa upload silinceye başarısız olur, ama veritabanı satırı `pdf_storage_path` değeriyle yine de yazılabilir — var olmayan bir dosyaya işaret eden bozuk bir kayıt oluşur.

**Fix (Story 7.1 smoke test öncesi):** Supabase Dashboard → Storage → "New bucket" ile `verfahrensdokumentation` adında **private** bir bucket oluştur. Bucket RLS politikası: Storage path'inin ilk segmenti (`(storage.foldername(name))[1]`) kullanıcının `tenant_id`'siyle eşleşmelidir. Erişim: `storage.from('verfahrensdokumentation').createSignedUrl(path, 3600)` ile signed URL üzerinden. Bu P3 kapsam dışında bırakıldı (P2 spike'ta açık kararlaştırıldı); Story 7.1 spike veya prepinde ele alınmalı.
